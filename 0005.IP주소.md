## 🌐 IP 주소 (Internet Protocol Address)  예시: 192.168.0.10

**IP 주소**는 인터넷에 연결된 모든 장치(컴퓨터, 스마트폰, 서버 등)에 부여되는 **고유한 식별 주소**이자, 네트워크를 통해 데이터를 주고받기 위해 사용되는 **논리적 주소** 체계입니다.

이 주소가 있어야 데이터가 출발지에서 목적지까지 정확하게 길을 찾아갈 수 있습니다.

---

### 1. IP 주소의 핵심 역할 (라우팅)

IP 주소의 가장 중요한 역할은 **라우팅(Routing)**입니다.

* **네트워크 식별:** IP 주소는 통신할 장치가 **어떤 네트워크 그룹**에 속해 있는지 알려줍니다.
* **경로 결정:** **공유기(Router)**와 같은 네트워크 장비는 IP 주소를 기반으로 패킷(데이터 묶음)이 목적지로 가기 위한 **최적의 경로**를 결정하고 전달합니다. (OSI 7계층 중 **3계층, 네트워크 계층**에서 사용됩니다.) 

[Image of the OSI 7-Layer Model illustrating Layer 3 (Network Layer) and the Router's role]


### 2. IP 주소의 구조와 종류

IP 주소는 크게 두 가지 버전이 사용되고 있으며, 각각의 구조와 특징이 다릅니다.

#### A. IPv4 (Internet Protocol version 4)

* **구조:** 8비트씩 4부분으로 나뉜 **32비트** 주소 체계입니다.
* **표현:** 각 부분을 점(.)으로 구분하여 `xxx.xxx.xxx.xxx` 형태로 표시합니다. (예: `192.168.0.1`)
* **한계:** 약 43억 개의 주소만을 표현할 수 있어, 인터넷 장치의 폭발적인 증가로 인해 **주소 고갈** 문제가 발생했습니다.

#### B. IPv6 (Internet Protocol version 6)

* **구조:** 16비트씩 8부분으로 나뉜 **128비트** 주소 체계입니다.
* **표현:** 각 부분을 콜론(:)으로 구분하여 16진수로 표시합니다. (예: `2001:0db8:85a3:0000:0000:8a2e:0370:7334`)
* **특징:** 사실상 무한대에 가까운 주소를 제공하며, 보안 및 서비스 품질(QoS) 기능이 강화되었습니다. 현재 IPv4의 한계를 극복하기 위해 점차적으로 전환되고 있습니다.

---

### 3. IP 주소의 분류 (공인 vs. 사설)

IP 주소는 사용 범위에 따라 크게 두 가지로 나뉩니다.

| 구분 | 공인 IP (Public IP) | 사설 IP (Private IP) |
| :--- | :--- | :--- |
| **특징** | 전 세계에서 유일하며, **인터넷 통신에 사용** | 특정 네트워크(집, 회사) 내에서만 사용되며, **중복 가능** |
| **할당 주체** | ISP (인터넷 서비스 제공업체) | 공유기(Router) (NAT 기술 사용) |
| **비고** | 서버, 공유기 외부에 할당 | 개인 PC, 스마트폰 등에 할당 |

> ### 💡 실전 팁: IP 주소는 논리적 주소
>
> IP 주소는 하드웨어에 고정된 **MAC 주소(물리적 주소)**와 달리, 사용자가 네트워크를 옮기거나 설정에 따라 **변경될 수 있는(논리적)** 주소라는 점을 이해하는 것이 중요합니다.

---
MAC 주소가 있는데도 굳이 IP 주소를 사용하는 이유에 대한 질문은 네트워크의 핵심 원리를 꿰뚫는 아주 좋은 질문입니다.

결론부터 말하면, **MAC 주소는 같은 동네(네트워크) 안에서 물건을 찾는 데 사용**되고, **IP 주소는 전 세계(인터넷)에서 그 동네까지 가는 길을 찾는 데 사용**됩니다.

---

## 🗺️ MAC 주소와 IP 주소의 역할 구분 (주소 체계의 분리)

두 주소의 역할과 계층을 분리함으로써, 인터넷은 거대한 규모에서도 효율적이고 유연하게 작동할 수 있습니다.

### 1. MAC 주소 (Media Access Control Address)

* **역할:** 같은 **로컬 네트워크(LAN)** 내에서 장치를 식별하는 데 사용됩니다.
* **성격:** **물리적 주소(Physical Address)**. 네트워크 인터페이스 카드(NIC)에 하드웨어적으로 할당되며 전 세계적으로 유일합니다. **2계층(데이터 링크 계층)**에서 사용됩니다.
* **비유:** **이웃집의 현관문 주소**나 **사람의 주민등록번호**와 같습니다. 고유하지만, 주소만으로는 그 사람이 어느 도시, 어느 구역에 사는지 바로 알 수 없습니다.

### 2. IP 주소 (Internet Protocol Address)

* **역할:** **다른 네트워크(원거리)** 간 통신(라우팅)을 위해 사용됩니다.
* **성격:** **논리적 주소(Logical Address)**. 소프트웨어적으로 설정되며, 네트워크 환경에 따라 변경될 수 있습니다. **3계층(네트워크 계층)**에서 사용됩니다.
* **비유:** **도시(네트워크)와 도로명**이 포함된 주소(예: 서울특별시 강남구). 이 주소 덕분에 **공유기(Router)**와 같은 장비가 패킷이 가야 할 큰 길(네트워크)을 결정할 수 있습니다.

---

## 🛣️ IP 주소가 반드시 필요한 3가지 핵심 이유

MAC 주소만으로는 전 세계적인 인터넷 통신이 불가능하며, IP 주소가 이 문제를 해결해 줍니다.

### 1. 라우팅 (Routing)의 효율성 확보

가장 중요한 이유입니다.

* **MAC 주소의 한계:** MAC 주소는 **계층적 구조가 없습니다.** 전 세계 80억 개 이상의 장치가 무작위의 고유한 MAC 주소를 가지고 있다면, 전 세계의 모든 공유기(Router)는 통신을 위해 이 수많은 주소들을 모두 저장하고 있어야 합니다. 이는 현실적으로 불가능하며 매우 비효율적입니다.
* **IP 주소의 강점:** IP 주소는 `네트워크 주소`와 `호스트 주소`로 나뉘는 **계층적 구조**를 가지고 있습니다.
    * 공유기는 패킷을 최종 목적지로 보내기 위해 **수많은 개별 장치의 주소(MAC)**를 알 필요 없이, **목적지가 속한 네트워크 그룹 주소(IP)**만 알면 됩니다. 마치 우체국이 개별 집 주소가 아닌 큰 지역 주소(시/도)만 보고 우편물을 분류하는 것과 같습니다. 

[Image of the OSI 7-Layer Model illustrating Layer 3 (Network Layer) and the Router's role]


### 2. 네트워크 환경의 유연성 제공

* MAC 주소는 하드웨어에 고정되지만, IP 주소는 **유동적**입니다.
* 장치가 네트워크를 이동하거나 접속 방식이 바뀌어도 (예: 와이파이에서 유선으로), IP 주소만 새롭게 할당받으면 바로 통신이 가능합니다. 이 유연성 덕분에 DHCP와 같은 기술로 주소를 효율적으로 관리할 수 있습니다.

### 3. 계층 분리를 통한 확장성

* **3계층(IP)과 2계층(MAC)의 분리** 덕분에, 우리는 하부의 물리적 통신 기술(2계층)이 바뀌더라도 (예: 이더넷이 Wi-Fi로 바뀌더라도) 상위의 인터넷 프로토콜(3계층)은 그대로 유지하면서 인터넷을 사용할 수 있습니다.
* 이러한 **모듈화(Modularization)** 덕분에 인터넷은 끊임없이 새로운 기술을 수용하며 확장할 수 있었습니다.

---

## 🗺️ IP 주소 (Internet Protocol Address) 체계 상세 분석

IP 주소는 네트워크 계층(Layer 3)에서 장치를 식별하고 데이터를 라우팅하는 논리적인 주소입니다. 이 주소는 단순한 숫자가 아닌, 효율적인 네트워크 관리를 위한 정교한 **체계(Addressing Scheme)**를 가지고 있습니다.

## 💻 현재 사용 중인 IP 주소 조회 

IP 주소 조회는 내부 네트워크의 **사설 IP**와 외부 인터넷에 노출되는 **공인 IP** 조회로 나눌 수 있습니다. 요청하신 대로 명령 프롬프트(CLI) 화면을 포함하여 각 조회 방법을 안내합니다.

-----

### 1\. 🌐 공인 IP 주소 조회 (외부망에서 나를 식별하는 주소)

공인 IP 주소는 인터넷 서비스 제공업체(ISP)가 사용자 네트워크에 할당한 주소이며, 외부 웹사이트를 통해 쉽게 확인할 수 있습니다.

  * **방법**: 웹 브라우저를 열고 구글 검색창에 \*\*"내 아이피 주소"\*\*라고 검색합니다.
  * **결과**: 검색 결과 페이지 상단에 현재 사용하고 있는 **공인 IP 주소**가 표시됩니다.

| 조회 서비스 결과 (예시) |
| :---: |
| **당신의 공인 IP 주소는 211.108.7.123 입니다.** |

### 2\. 🖥️ Windows 장치의 사설 IP 주소 조회

Windows 장치(PC, 서버)에 할당된 IP 주소(사설 IP)는 `ipconfig` 명령어를 통해 확인합니다.

```cmd
C:\Users\ITExpert> ipconfig
```

#### 🔸 출력 (예시)

```
C:\Users\ITExpert> ipconfig

Windows IP 구성


이더넷 어댑터 이더넷:

   연결별 DNS 접미사 . . . . : 
   **IPv4 주소 . . . . . . . . : 192.168.10.50** <-- 이 장치의 사설 IP
   서브넷 마스크 . . . . . . : 255.255.255.0
   **기본 게이트웨이 . . . . . : 192.168.10.1** <-- 라우터(공유기) 주소
```

  * **확인 항목**: 출력에서 현재 연결된 어댑터(이더넷 또는 Wi-Fi) 항목의 \*\*`IPv4 주소`\*\*를 확인합니다.

### 3\. 🐧 Linux 장치의 사설 IP 주소 조회

Linux 장치에서는 최신 명령어인 `ip a` (ip address show)를 사용하여 IP 주소를 확인합니다.

```bash
$ ip a
```

#### 🔸 예상 출력 (예시)

```bash
$ ip a
1: lo: ...
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether 00:1a:2b:3c:4d:5e brd ff:ff:ff:ff:ff:ff
    **inet 192.168.10.50/24** brd 192.168.10.255 scope global dynamic eth0  <-- 이 장치의 사설 IP
       valid_lft 85310sec preferred_lft 85310sec
    inet6 fe80::21a:2bff:fe3c:4d5e/64 scope link 
       valid_lft forever preferred_lft forever
```

  * **확인 항목**: 활성 인터페이스(예: `eth0`) 항목에서 **`inet`** 뒤에 표시된 주소와 CIDR 프리픽스(`/24`)를 확인합니다.

### 4\. 🔍 특정 공인 IP의 할당 정보 조회 (WHOIS)

다른 사람의 공인 IP 주소가 어느 국가, 어느 기관에 할당되었는지 확인하려면 WHOIS 검색을 사용합니다.

```cmd
C:\Users\ITExpert> nslookup 8.8.8.8
```

  * **명령어 예시**: `nslookup` 명령어는 DNS 정보를 조회하는 데 쓰이지만, IP 주소를 인자로 넣어 해당 IP의 도메인이나 할당 기관 정보를 간접적으로 확인할 수 있습니다. 더 정확한 할당 정보는 **KISA(한국인터넷진흥원) 후이즈 검색**과 같은 전문 서비스를 이용해야 합니다

### 1. 🔑 IP 주소의 구성 요소 및 체계 개요

IP 주소 체계는 크게 두 부분으로 나뉘어 네트워크의 주소와 해당 네트워크 내 호스트의 주소를 구분합니다.

| 구성 요소 | 설명 | 역할 |
| :---: | :--- | :--- |
| **IP 주소** (예: 192.168.1.10) | 장치를 고유하게 식별하는 논리 주소 | 데이터의 **목적지**를 지정 |
| **서브넷 마스크** (예: 255.255.255.0) | IP 주소에서 **네트워크 ID**와 **호스트 ID**를 구분하는 마스크 | **로컬 네트워크** 범위를 지정 |
| **네트워크 ID** (Network ID) | 통신망 자체를 식별하는 주소 | 라우터가 **경로**를 결정하는 기준 |
| **호스트 ID** (Host ID) | 해당 네트워크 내의 특정 장치(호스트)를 식별하는 주소 | 네트워크 ID 내에서 **장치를 구분** |

### 2. 📝 IPv4 주소 체계: 클래스풀 및 CIDR

#### 🔸 클래스풀 주소 체계 (Classful Addressing)
## 🏷️ IPv4 주소 클래스별 특징 및 구조

요청하신 형식에 맞추어 IPv4 주소의 클래스별 특징을 **첫 번째 옥텟의 2진수 패턴**과 **서브넷 마스크**를 포함하여 정리했습니다. 현재는 CIDR(클래스리스 라우팅)이 표준이지만, 이 표는 주소 체계의 기본 이해를 돕습니다.

| 클래스 | 1번 옥텟 (2진수 시작 패턴) | 범위 (10진수) | 네트워크 비트 | 서브넷 마스크 (기본) | 특징 (규모, 용도) | 사용 가능 호스트 수 ($2^{\text{H}}-2$) |
| :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| **A** | **0**xxxxxxx | 1 ~ 126 | 8 비트 | 255.0.0.0 | **대규모** 네트워크 (최대 호스트 수) | 약 1,600만 개 |
| **B** | **10**xxxxxx | 128 ~ 191 | 16 비트 | 255.255.0.0 | **중규모** 네트워크 (균형 잡힘) | 약 65,000개 |
| **C** | **110**xxxxx | 192 ~ 223 | 24 비트 | 255.255.255.0 | **소규모** 네트워크 (최대 네트워크 수) | 254개 |
| **D** | **1110**xxxx | 224 ~ 239 | N/A | N/A | **멀티캐스트** 전용 (그룹 통신) | N/A |
| **E** | **1111**xxxx | 240 ~ 255 | N/A | N/A | **연구/예비용** | N/A |

---

### 🔎 표 해설 및 주요 개념

* **네트워크 비트**: IP 주소 32비트 중 **네트워크 ID**를 정의하는 비트의 길이입니다.
    * 예: 클래스 C는 24비트(3옥텟)가 네트워크 ID를 정의하며, $2^{24}$개의 네트워크를 만들 수 있습니다.
* **호스트 비트**: IP 주소 32비트 중 **호스트 ID**를 정의하는 비트의 길이입니다.
    * 예: 클래스 C는 8비트가 호스트 ID이므로, $2^8 = 256$개의 주소를 가질 수 있습니다.
* **사용 가능 호스트 수**:
    * 각 네트워크에서는 **네트워크 주소(호스트 ID가 모두 0)**와 **브로드캐스트 주소(호스트 ID가 모두 1)** 두 개는 특수 용도로 사용되므로, 실제 장치에 할당 가능한 주소는 $2^{\text{호스트 비트}} - 2$ 개입니다.
    * 클래스 A는 호스트 비트가 24개이므로 $2^{24}-2$입니다.
* **서브넷 마스크**: 네트워크 비트에 해당하는 부분을 **1**로 채우고, 호스트 비트에 해당하는 부분을 **0**으로 채운 값입니다. 이 마스크를 IP 주소와 **AND 연산**하면 네트워크 ID를 추출할 수 있습니다. (예: 클래스 C는 24비트가 1이므로 `255.255.255.0`)

### 💡 클래스 D 주소의 특별함
클래스 D 주소는 처음 네 비트가 **1110**으로 시작하며, 데이터를 **특정 그룹**에게만 전송하는 **멀티캐스트** 전용입니다. 따라서 네트워크/호스트 구분이 의미가 없으며, 라우팅 시 그룹 주소로 처리됩니다.



#### 🔹 클래스리스(클래스 없는) 주소 체계 (CIDR, Classless Inter-Domain Routing)


**클래스 없는 도메인 간 라우팅 (CIDR, Classless Inter-Domain Routing)**은 기존의 **클래스풀(Classful)** IP 주소 체계가 가진 비효율성과 IP 주소 고갈 문제를 해결하기 위해 1993년에 도입된 현대적인 IP 주소 할당 및 라우팅 방식입니다.

CIDR은 IP 주소에서 네트워크와 호스트를 구분하는 경계를 **고정된 클래스 경계(8, 16, 24비트)**에 얽매이지 않고, **유연하게** 정의할 수 있게 해줍니다.

---

### 1. 🔑 CIDR의 핵심 원리: Prefix Length

CIDR의 가장 중요한 특징은 IP 주소 뒤에 슬래시(`/`)와 함께 **Prefix Length** (또는 네트워크 프리픽스 길이)를 붙여 네트워크 ID의 길이를 직접 표시하는 것입니다.

* **표기 형식**: `IP 주소/Prefix Length`
* **Prefix Length의 의미**: 32비트 IP 주소 중 **네트워크 ID로 사용되는 비트의 수**를 나타냅니다. (이는 곧 서브넷 마스크에서 `1`의 개수와 같습니다.)
* **예시**:
    * `192.168.1.0/24` $\rightarrow$ 처음 24비트가 네트워크 ID이고, 나머지 8비트가 호스트 ID입니다.
    * `10.0.0.0/8` $\rightarrow$ 처음 8비트만 네트워크 ID입니다.

| IP 주소 | CIDR 표기 | 서브넷 마스크 | 네트워크 ID (N) | 호스트 ID (H) |
| :---: | :---: | :---: | :---: | :---: |
| 192.168.1.0 | **/24** | 255.255.255.0 | 24비트 | 8비트 |
| 172.16.0.0 | **/16** | 255.255.0.0 | 16비트 | 16비트 |
| 192.168.1.32 | **/27** | 255.255.255.224 | 27비트 | 5비트 |

---

### 2. 💡 CIDR이 해결한 두 가지 문제

#### 🔸 1. IP 주소 낭비 문제 해결 (VLSM)
클래스풀 방식에서는 필요한 호스트 수에 관계없이 클래스 A, B, C의 고정된 크기만 사용할 수 있어 주소 낭비가 심했습니다.

* **해결책**: **가변 길이 서브넷 마스크 (VLSM, Variable Length Subnet Mask)** 구현
    * CIDR을 통해 네트워크 관리자는 필요한 호스트 수에 맞춰 **서브넷의 크기를 자유롭게** 조정할 수 있습니다.
    * 예를 들어, 호스트 10개만 필요한 링크에는 `/28` (호스트 14개)을 할당하고, 호스트 500개가 필요한 네트워크에는 `/23` (호스트 510개)을 할당하여 주소 낭비를 최소화합니다.

#### 🔹 2. 라우팅 테이블 비대화 문제 해결 (Supernetting)
인터넷이 확장되면서 라우터가 처리해야 할 네트워크 경로 정보가 폭발적으로 증가했습니다.

* **해결책**: **슈퍼넷팅 (Supernetting)** 또는 경로 집계 (Route Summarization)
    * CIDR을 사용하면 여러 개의 작은 네트워크를 하나의 **더 큰 블록**으로 묶어 라우터에 알릴 수 있습니다.
    * **예시**: `200.1.0.0/24`, `200.1.1.0/24`, `200.1.2.0/24`, `200.1.3.0/24` 네 개의 클래스 C 네트워크를 하나의 **`200.1.0.0/22`** 경로로 통합하여 라우팅 테이블에 등록할 수 있습니다.
    * 이로 인해 라우팅 테이블의 크기가 줄어들고 라우터의 효율이 증가합니다.

---

### 3. 🧠 CIDR을 사용한 서브넷팅 계산 (예시)

CIDR은 네트워크 관리자가 원하는 크기로 네트워크를 나누는 **서브넷팅**을 가능하게 합니다.

**요구사항**: 클래스 C 네트워크 `192.168.1.0/24`를 사용하여 4개의 서브넷을 생성하고 싶습니다.

1.  **필요한 비트 계산**: 4개의 서브넷을 만들기 위해서는 $2^n \ge 4$를 만족하는 $n$이 필요합니다. $\mathbf{2^2 = 4}$이므로, **호스트 비트에서 2비트**를 빌려와야 합니다.
2.  **새로운 Prefix Length**: 기존 `/24`에 2비트를 추가하면 **`/26`**이 됩니다.
3.  **새로운 서브넷 마스크**: `/26`은 1이 26개 있는 마스크입니다.
    * $255.255.255$. $\underbrace{11000000}_{26\text{th bit}}$
    * $11000000_2$는 10진수로 $128 + 64 = \mathbf{192}$입니다.
    * 새로운 서브넷 마스크는 **255.255.255.192**가 됩니다.
4.  **서브넷 크기**: 호스트 ID로 남은 비트는 $32 - 26 = 6$비트입니다.
    * 각 서브넷은 $2^6 = 64$개의 주소를 가지며, 사용 가능 호스트 수는 $64-2 = 62$개입니다.

| 서브넷 번호 | 네트워크 주소 (CIDR) | 호스트 범위 | 브로드캐스트 주소 |
| :---: | :---: | :---: | :---: |
| **0** | **192.168.1.0/26** | 192.168.1.1 ~ 192.168.1.62 | 192.168.1.63 |
| **1** | **192.168.1.64/26** | 192.168.1.65 ~ 192.168.1.126 | 192.168.1.127 |
| **2** | **192.168.1.128/26** | 192.168.1.129 ~ 192.168.1.190 | 192.168.1.191 |
| **3** | **192.168.1.192/26** | 192.168.1.193 ~ 192.168.1.254 | 192.168.1.255 |

---

### 3. 🛡️ 특수 IP 주소 및 사설 IP 체계

특정 목적을 위해 예약되거나 공중망에서 사용되지 않는 IP 주소들이 있습니다.

#### 🔸 사설 IP 주소 (Private IP Address)

인터넷에 직접 연결되지 않고, NAT(Network Address Translation) 장비를 통해 공인 IP로 변환되어야만 외부 통신이 가능한 주소입니다.

| 클래스 | 주소 범위 | 설명 |
| :---: | :--- | :--- |
| **A** | `10.0.0.0` ~ `10.255.255.255` (`10.0.0.0/8`) | 대형 사설망에 사용 |
| **B** | `172.16.0.0` ~ `172.31.255.255` (`172.16.0.0/12`) | 중형 사설망에 사용 |
| **C** | `192.168.0.0` ~ `192.168.255.255` (`192.168.0.0/16`) | 일반 가정 및 소규모 사무실에 가장 많이 사용 |

#### 🔹 특수 목적 IP 주소

| 주소 | 설명 | 역할 |
| :---: | :--- | :--- |
| **네트워크 주소** | 호스트 ID가 모두 0인 주소 | 해당 네트워크 자체를 대표하는 주소 |
| **브로드캐스트 주소** | 호스트 ID가 모두 1인 주소 | 해당 네트워크 내의 **모든 호스트**에게 데이터를 전송할 때 사용 |
| **루프백 주소** | `127.0.0.1` (`127.0.0.0/8`) | 자기 자신에게 통신할 때 사용하는 주소 (**장치 테스트용**) |
| **APIPA** | `169.254.0.0/16` | DHCP 서버로부터 IP를 할당받지 못했을 때 장치 스스로 임시로 설정하는 주소 |

## ❓ 클래스 A 주소 대역의  특수 목적 대역
 클래스 A 범위 내에는 특정 목적을 위해 사용이 제한되거나 약속된 특별한 주소 대역들이 존재합니다.

### 1. 🌐 공인 IP 주소로서의 사용 (88.x.x.x 대역)

클래스 A 범위에 속하는 대부분의 주소는 **공인 IP 주소**로 간주되어 전 세계 인터넷에서 사용됩니다.


* **할당 관리**: 과거에는 클래스 A 주소 전체를 한 기업이 독점할 수 있었지만, 현재는 **CIDR (Classless Inter-Domain Routing)** 체계 하에서 필요에 따라 `/8`보다 작은 단위(예: `/16`, `/24`)로 통신 사업자에게 분배되어 효율적으로 사용됩니다.

---

### 2. 🔒 클래스 A 내의 약속된 특수 대역

클래스 A 범위(1 ~ 126) 내에서 특별한 목적으로 예약되거나 사용이 제한되는 중요한 대역이 세 가지 있습니다.

#### 🔸 0.0.0.0/8 (0.x.x.x 대역)
* **목적**: **예약됨 (Reserved)**
* **설명**: 네트워크 자체를 의미하거나 **'이 네트워크'**를 의미하는 데 사용되는 주소입니다. 호스트가 자신의 IP 주소를 아직 모를 때 사용하거나, 라우팅 테이블에서 기본 경로(Default Route)를 설정할 때 사용됩니다.

#### 🔹 10.0.0.0/8 (10.x.x.x 대역)
* **목적**: **사설 IP (Private IP)**
* **설명**: **RFC 1918**에 의해 **공인 인터넷에서 사용이 금지**된 사설 주소 대역 중 가장 큰 규모의 대역입니다. 기업의 대규모 내부망이나 데이터센터에서 호스트 장치에 IP 주소를 할당할 때 사용됩니다. 이 주소는 외부 인터넷과 통신하기 위해 **NAT**를 거쳐야 합니다.

#### 🔸 127.0.0.0/8 (127.x.x.x 대역)
* **목적**: **루프백 (Loopback)**
* **설명**: 장치 **자기 자신**에게 통신할 때 사용하는 특수한 예약 대역입니다. 실제 네트워크 카드나 케이블을 통하지 않고, TCP/IP 소프트웨어 스택이 정상 작동하는지 테스트하는 데 사용됩니다.
    * 대표적인 주소: **127.0.0.1**

---

### 3. 📝 클래스 A 주소 체계 요약

클래스 A 전체 범위 중에서 실제로 공인 통신망에서 활용 가능한 대역은 특수 목적 대역을 제외한 나머지 부분입니다.

| IP 주소 범위 (첫 옥텟) | 용도 | 특징 |
| :---: | :---: | :--- |
| **1 ~ 9** | 공인 IP | 일반 인터넷 사용 |
| **10** | 사설 IP | **내부망 전용 (NAT 필수)** |
| **11 ~ 126** | 공인 IP | 일반 인터넷 사용  |
| **127** | 루프백 | 자기 진단용 (내부 테스트) |

### 4. 🚀 IPv6 주소 체계

IPv4의 주소 고갈 문제를 해결하기 위해 등장했습니다.

* **표기법**: 16비트씩 여덟 부분으로 나누어 **콜론(:)**으로 구분하고 16진수로 표기합니다.
    * **축약 규칙**: `0000`처럼 연속되는 0은 생략 가능하며, 연속된 0의 블록은 한 번에 `::`으로 축약 가능합니다. (예: `2001:db8:0:0:0:0:0:1` $\rightarrow$ `2001:db8::1`)
* **주소 타입**:
    * **유니캐스트**: 특정 하나의 장치에 할당 (가장 일반적)
    * **멀티캐스트**: 특정 그룹에 속한 다수 장치에 전송
    * **애니캐스트**: 동일 주소를 가진 가장 가까운 장치에 전송

IP 주소 체계를 이해하는 것은 서브넷을 설계하고, 효율적인 라우팅 정책을 수립하며, IP 고갈 및 보안 문제에 대응하는 첫걸음입니다.

---
## 🚀 IPv6 도입 현황 및 체계의 특징

IPv6 주소 체계가 길고 복잡하게 느껴지는 것은 당연합니다. **128비트**라는 거대한 크기 때문에 기존의 **IPv4(32비트)**처럼 사람이 쉽게 외우거나 계산하기는 어렵습니다. 하지만 IPv6는 현재 인터넷의 미래를 위해 필수적으로 도입되고 있으며, 점차 사용량이 늘어나고 있습니다.

### 1. 🌐 IPv6 현재 도입 현황

IPv6를 "본 적이 없다"고 느끼는 이유는 주로 **최종 사용자 단말(PC, 스마트폰)**에서는 IPv4가 여전히 기본으로 설정되어 있거나, **IPv4와 IPv6를 함께 사용하는 전환 기술**이 적용되어 있기 때문입니다.

* **글로벌 사용량 증가**: 주요 인터넷 서비스 제공업체(ISP), 모바일 네트워크, 대형 콘텐츠 제공업체(Google, Facebook, Netflix 등)는 이미 IPv6를 **필수적으로 사용**하고 있습니다. 많은 국가와 기업에서 IPv6 전환율이 50%를 넘어서고 있습니다.
* **모바일 네트워크**: 스마트폰과 같은 모바일 장치는 새로운 IP 주소를 효율적으로 할당하기 위해 상당 부분 **IPv6를 기반**으로 통신하고 있습니다.
* **대규모 네트워크**: 데이터센터, 클라우드 환경(AWS, Azure 등), 그리고 5G 코어망은 주소 고갈 문제를 피하고 성능을 최적화하기 위해 **IPv6를 표준**으로 채택하고 있습니다.

> **결론**: **사용자의 눈에 잘 띄지 않을 뿐**, 인터넷의 **백본(Backbone)망**과 **클라우드 인프라**에서는 IPv6가 활발하게 사용되고 있으며, 일반 사용자의 접속도 점진적으로 증가하고 있습니다.

---

### 2. 🔢 IPv6 주소 체계가 긴 이유와 장점

IPv6 주소가 128비트로 길어진 이유는 단순히 주소 고갈 문제를 해결하는 것을 넘어, **라우팅 효율성**과 **자동 설정** 등 여러 이점을 제공하기 위함입니다.

#### 🔸 길고 복잡한 주소가 주는 이점

| 장점 | 설명 |
| :--- | :--- |
| **주소 고갈 문제 해소** | IPv4의 43억 개를 넘어선 **$3.4 \times 10^{38}$** 개라는 방대한 주소 공간을 확보했습니다. |
| **효율적인 라우팅** | IPv6는 주소의 절반인 **64비트**를 **네트워크 프리픽스(Prefix)**로 사용하고, 나머지 64비트를 **인터페이스 식별자(Interface ID)**로 사용하도록 **규격화**되어 있습니다. 라우터가 주소 앞부분(프리픽스)만 보고 쉽게 경로를 결정할 수 있어 **라우팅 테이블의 크기를 줄이고 처리 속도를 높입니다.** |
| **자동 주소 설정 (SLAAC)** | 호스트가 **DHCP 서버 없이** 자신의 MAC 주소 기반으로 64비트 인터페이스 식별자를 만들어 IP 주소를 **자동으로 설정**할 수 있습니다 (**Stateless Address Autoconfiguration**). 관리의 복잡성을 줄여줍니다. |
| **보안 강화 (IPsec 내장)** | IPv6 헤더에는 **IPsec(IP Security)** 기능이 기본적으로 내장되어 있어, 패킷의 인증 및 암호화 기능을 쉽게 구현할 수 있습니다. |

#### 🔹 복잡한 주소를 다루는 실용적인 방법

>실제로는 이 긴 주소를 사람이 직접 다루거나 외울 필요가 거의 없습니다.
>
>1.  **축약 규칙**: 연속된 `0`을 `::`으로, 선행 `0`을 생략하는 등의 **축약 규칙**을 통해 실제 사용 및 표기 시 길이를 줄입니다. (예: `2001:0db8::1`)
>2.  **DNS 사용**: IPv4와 마찬가지로, 사용자는 **도메인 이름(DNS)**을 사용하고, 시스템이 내부적으로 IPv6 주소를 처리합니다.
>3.  **자동 생성**: 대부분의 IPv6 주소는 장치의 MAC 주소 기반으로 **자동 생성**되므로, 관리자가 일일이 할당하고 외울 필요가 없습니다.

5.  ## 📡 D 클래스 주소와 멀티캐스트 통신 원리

**클래스 D IP 주소**는 일반적인 유니캐스트(1:1) 통신이나 브로드캐스트(1:모두) 통신을 위해 사용되지 않고, 오직 **멀티캐스트(Multicast)** 통신만을 위해 예약되어 있습니다.

D 클래스 주소는 **224.0.0.0**부터 **239.255.255.255**까지의 범위를 가지며, 이 주소는 **특정 수신 그룹**을 식별하는 용도로 사용됩니다.



### 1. 👥 D 클래스 주소의 역할 (멀티캐스트 그룹 주소)

클래스 D 주소는 **'특정 관심사를 가진 그룹'**의 주소입니다. 데이터를 받는 개별 장치의 주소가 아니라, **데이터를 수신하겠다고 등록한 장치들**이 공유하는 **논리적인 주소**입니다.

1.  **그룹 식별**: D 클래스 주소 하나가 하나의 멀티캐스트 그룹을 대표합니다. 예를 들어, `224.0.0.5`는 OSPF 라우팅 프로토콜을 사용하는 모든 라우터를 의미하는 그룹 주소입니다.
2.  **데이터 전송**: 송신자(Source)는 데이터를 보낼 때, 목적지 IP 주소로 이 **클래스 D 주소**를 지정합니다.
3.  **수신자 등록**: 데이터를 받고자 하는 장치들은 **IGMP(Internet Group Management Protocol)**라는 프로토콜을 사용하여 스위치와 라우터에 해당 멀티캐스트 그룹(`224.x.x.x`)에 **가입**하겠다는 의사를 전달합니다.

### 2. 🏷️ D 클래스 주소의 작동 방식

라우터나 스위치는 D 클래스 주소를 목적지로 하는 패킷을 받으면, 유니캐스트처럼 하나의 경로로 보내거나 브로드캐스트처럼 모든 곳에 보내지 않고, **특정 그룹에 가입한 경로와 포트로만 복사하여 전송**합니다.

#### 🔸 MAC 주소와의 연동
멀티캐스트 통신을 효율적으로 수행하기 위해, 클래스 D IP 주소는 2계층(데이터 링크)에서 사용되는 **멀티캐스트 MAC 주소**와 일대일로 매핑됩니다.

* **멀티캐스트 MAC 주소의 구조**: 멀티캐스트 MAC 주소는 항상 `01:00:5E:xx:xx:xx` 형태로 시작합니다.
* **매핑 과정**: 스위치나 라우터는 클래스 D IP 주소의 하위 23비트를 이 MAC 주소의 하위 23비트에 대응시켜 사용합니다. 즉, 라우터는 D 클래스 IP 주소를 보고 해당 **멀티캐스트 MAC 주소**를 결정한 후, 이를 사용하여 프레임을 전송합니다.

이 매핑 덕분에, 스위치는 3계층(IP) 주소를 분석하지 않고도 2계층(MAC) 주소만으로 그룹에 가입된 장치가 연결된 포트로만 데이터를 정확히 전달할 수 있습니다 (**IGMP Snooping** 기능).

### 3. 📺 주요 사용 예시

D 클래스 주소는 네트워크의 대역폭을 낭비하지 않으면서 동시에 다수의 장치에게 동일한 정보를 효율적으로 전달해야 할 때 사용됩니다.

* **IPTV 및 비디오 스트리밍**: 실시간 비디오 방송을 시청하는 구독자 그룹에게만 영상을 전송합니다.
* **라우팅 프로토콜**:
    * **OSPF**: 라우터 간에 정보를 교환할 때 `224.0.0.5`와 `224.0.0.6` 주소를 사용합니다.
    * **RIPv2**: `224.0.0.9` 주소를 사용하여 정보를 전송합니다.
* **분산 데이터 시스템**: 다수의 서버나 클라이언트에게 동시에 업데이트 정보를 보내야 하는 환경.

## 👻 Bogon IP (보고온 IP) 주소 설명

**Bogon IP (보고온 IP)** 주소는 인터넷 라우팅 환경에서 **절대 라우팅되거나 사용되어서는 안 되는(Should Never Be Seen)** IP 주소 블록을 일컫는 비공식적인 용어입니다.

이 주소들은 인터넷 표준에 의해 **예약(Reserved)**되어 있거나, **할당되지 않은(Unallocated)** 주소이거나, 혹은 **사설망 전용**으로 지정되어 공인 인터넷망에서는 존재하지 않아야 하는 주소들입니다.

### 1. 🔍 Bogon IP의 정의 및 특징

'Bogon'이라는 단어는 '허위의', '가짜의'라는 뜻을 가진 속어에서 유래했습니다. 네트워크 보안 관점에서는 다음과 같은 특징을 가집니다.

* **비할당 주소 (Unallocated)**: IANA나 RIR(지역 인터넷 레지스트리)에 의해 아직 어떤 조직이나 ISP에도 할당되지 않은 주소 대역입니다.
* **예약 주소 (Reserved)**: 특정 기술적 목적(예: 루프백, 테스트, 멀티캐스트 등)을 위해 표준으로 예약되어 일반 통신용으로 사용되지 않는 주소입니다.
* **사설 주소 (Private)**: 내부 네트워크에서만 사용하도록 RFC 1918에 지정된 주소 대역입니다.

### 2. 🛡️ Bogon 필터링의 중요성 (보안 관점)

네트워크 관리자들은 라우터와 방화벽에 **Bogon 필터(Bogon Filter)**를 적용하여 외부로부터 들어오거나 내부에서 외부로 나가는 Bogon IP 트래픽을 차단합니다. 이는 네트워크 보안에 매우 중요합니다.

| 이유 | 설명 |
| :---: | :--- |
| **스푸핑 방지** | 공격자가 Bogon IP 주소를 **출발지 주소로 위조(IP Spoofing)**하여 DoS/DDoS 공격을 시도하는 것을 차단합니다. |
| **라우팅 테이블 정리** | 불필요하거나 유효하지 않은 경로 정보를 라우팅 테이블에서 제거하여 라우터의 효율성을 높입니다. |
| **경계 보호** | 사설 IP 주소가 실수로 공인 인터넷망으로 유출되는 것을 방지합니다. |

---

### 3. 📝 대표적인 Bogon IP 주소 대역 (IPv4)

Bogon 주소 목록은 RIR의 할당 정책에 따라 수시로 변동되지만, 다음 대역들은 일반적으로 항상 Bogon으로 간주되어 필터링됩니다.

| Bogon IP 대역 | CIDR 표기 | 비고 |
| :---: | :---: | :--- |
| **0.0.0.0/8** | 0.x.x.x | 예약됨. '이 네트워크'를 의미함. |
| **10.0.0.0/8** | 10.x.x.x | **사설 IP 대역 (클래스 A)**. |
| **100.64.0.0/10** | 100.64 ~ 100.127 | Carrier-Grade NAT (CGN) 전용으로 예약됨. |
| **127.0.0.0/8** | 127.x.x.x | **루프백(Loopback)** 주소. |
| **169.254.0.0/16** | 169.254.x.x | **APIPA (Automatic Private IP Addressing)** 주소. |
| **172.16.0.0/12** | 172.16 ~ 172.31 | **사설 IP 대역 (클래스 B)**. |
| **192.0.0.0/24** | 192.0.0.x | IANA 예약 대역. |
| **192.168.0.0/16** | 192.168.x.x | **사설 IP 대역 (클래스 C)**. |
| **224.0.0.0/4** | 224 ~ 239.x.x.x | **멀티캐스트(D 클래스)** 주소. |
| **240.0.0.0/4** | 240 ~ 255.x.x.x | **예약 및 연구용(E 클래스)** 주소. |

### 4. 🔄 최신 Bogon 목록 관리

Bogon IP 목록은 인터넷 주소 할당이 진행됨에 따라 정기적으로 업데이트됩니다.

* 네트워크 보안 전문가는 RIR(예:  APNIC, ARIN, RIPE NCC)에서 제공하는 최신 **비할당 주소 목록**을 주기적으로 다운로드하여 방화벽 및 라우터의 필터링 규칙을 갱신합니다.

Bogon IP 필터링은 네트워크의 '위생'을 유지하고 외부 공격 경로를 차단하는 기본적인 보안 실무입니다.
